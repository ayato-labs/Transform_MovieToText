# モバイル移行調査報告書:オンデバイスAI文字起こしアプリの構築

UIを共通化できるのであれば、それもCommonに配置したほうが良いのでは?UIを分割するっていう機能以外に手を出す余力があれば、セパレートするって形で。

## 1. 概要
本プロジェクト(Transform_MovieToText)をAndroidモバイルプラットフォームへ移行するための技術的実行可能性、推奨SDK、およびアーキテクチャ設計についての調査結果をまとめます。デスクトップ版の「ローカルLLM + Whisper」というコンセプトを維持しつつ、モバイル特有の低リソース・バッテリー制約に最適化する戦略を提案します。

---

## 2. 推奨モバイルSDK・技術スタック (2026年版)

### A. LLM推論エンジン (要約・要点抽出)
デスクトップ版のOllama/Gemmaに代わる、Androidネイティブの推論エンジンとして以下を採用します。

*   **MediaPipe LLM Inference API / LiteRT-LM**:
    *   **特徴**: Google公式のオンデバイスLLM推論フレームワーク。Gemma 3などの最新モデルに最適化されています。
    *   **メリット**: スマホのGPUおよびNPU(Neural Processing Unit)を直接活用でき、CPU負荷を劇的に抑えつつ高速推論が可能。
    *   **モデル**: `Gemma 3 2B` を 4-bit 量子化(GGUF等)して使用することを推奨。
*   **llama.cpp (via JNI)**:
    *   既存のデスクトップ資産(GGUFモデル等)を流用しやすい選択肢。

### B. 音声認識エンジン (文字起こし)
Faster-Whisperの代わりに、モバイル向けに極限まで最適化されたエンジンを採用します。

*   **whisper.cpp (Android JNI)**:
    *   **特徴**: C++によるWhisperのフル再実装。AndroidのNEON/GPU命令セットを活用。
    *   **運用**: 16bit量子化または8bit量子化モデルを使用し、リアルタイムに近い速度を実現。
*   **Google AI Edge / MediaPipe Speech-to-Text**:
    *   デバイス標準のNPUアクセラレーションが効くため、最もバッテリー効率が良い。

### C. UIフレームワーク
*   **Flet (Python/Flutter)**:
    *   既存のPythonコードを最大限再利用。
    *   **課題**: PythonからAndroidネイティブAI SDKを叩くには、`Pyjnius` というブリッジライブラリが必要。
*   **Flutter (Native)**:
    *   パフォーマンスと安定性を重視する場合、UIもFlutter(Dart)で再構築。

---

## 3. モバイル最適化アーキテクチャ (プロトタイプ設計)

### 構成案:ハイブリッド推論モデル
1.  **AI Engine Manager (Kotlin/C++)**:
    *   モデルのロード、NPUの確保、リアルタイム音声処理を担当。UIスレッドから完全に切り離してバックグラウンドで実行。
2.  **Logic Layer (Python/Flet)**:
    *   プロンプトの構成、履歴DB(SQLite)の管理、UI表示を担当。
3.  **ブリッジ層 (JNI/Pyjnius)**:
    *   Python側から「文字起こし開始」「要約実行」などのコマンドをネイティブエンジンに発行。

### 低リソース端末向けの調整
*   **モデル常駐の排除**: メモリ消費を抑えるため、使用時のみモデルをロード。
*   **量子化の徹底**: `Gemma 3 2B` (1Bも検討) のINT4量子化モデルを採用。
*   **サンプリングレート固定**: 音声入力はWhisperの仕様に合わせて 16kHz に固定。

---

## 4. 開発ロードマップ案

1.  **Phase 1: 検証 (POC)**
    *   MediaPipe LLM Inference APIを使用して、Android端末上で `Gemma 3 2B` が文字を生成できるか確認。
2.  **Phase 2: 統合**
    *   `whisper.cpp` のAndroid Demoをベースに、音声データの受け渡しを実装。
3.  **Phase 3: Fletモバイルビルド**
    *   FletをAndroid向けにビルドし、`Pyjnius` 経由で Phase 2 のエンジンと通信。

---

## 5. 結論と提言
現在のデスクトップ版の設計思想(プライバシーファースト・ローカルAI)は、**「MediaPipe (Gemma 3) + whisper.cpp」**という組み合わせでAndroid上でも十分に実現可能です。

特に、**「モデル選択を排除し、特定の軽量モデルに最適化する」**というユーザー提案は、モバイルAIにおける「勝ち筋」です。専用のアクセラレーション(NPU)を最大限利用するようにバイナリを調整することで、市販のスマホでも十分実用的な速度の「専属AI記者」アプリが完成するはずです。
