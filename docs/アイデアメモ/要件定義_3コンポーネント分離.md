# Ayato AI エコシステム: 要件定義書 (3コンポーネント分離版)

> **目的**: ポータル(Web)、ゲートウェイ(Server)、デスクトップアプリ(Client) の
> 責任境界を明確にし、「何を・何で実装するか」「何を実装しないか」を定義する。

---

## A. Ayato Studio Portal (Web)

**リポジトリ**: `ayato_studio_portal`
**技術スタック(既存)**: Next.js 16 / TypeScript / Supabase / Tailwind CSS v4 / Framer Motion
**ホスティング**: Cloudflare Pages (予定)
**現状**: AI レポート表示、ブログ、LP として稼働中。Supabase クライアントと Auth Callback は実装済み。

### A-1. 機能要件: 「何を・何で実装するか」

| 機能 | 使用技術 | 説明 |
| :--- | :--- | :--- |
| **Google ログイン** | **Supabase Auth** (Google OAuth) | ユーザーアカウントの作成とセッション管理。 |
| **メールログイン** | **Supabase Auth** (Magic Link) | Google アカウントを持たないユーザー向け。 |
| **プロダクトカタログ表示** | **Next.js ページ** | Transcriber, Reporter 等の商品一覧と LP。静的ページ。 |
| **サブスクリプション購入** | **Stripe Checkout** (リダイレクト型) | ユーザーを Stripe のホスト画面へ送り、決済を完了させる。 |
| **契約状態の DB 同期** | **Stripe Webhook** + **Supabase DB** | 決済完了/更新/解約の通知を受け取り、`organizations.plan_type` を更新。 |
| **支払い情報の管理** | **Stripe Customer Portal** (リダイレクト型) | カード変更、領収書閲覧、プラン変更を Stripe に委譲。 |
| **API トークン発行 UI** | **Supabase DB** (`api_tokens` テーブル) | ユーザーが自分のトークンを生成・閲覧・破棄できるマイページ。 |
| **利用状況の表示** | **Supabase DB** (`organizations` テーブル) | API コール数、プラン上限の表示（既存の `fetchOrganizationMetrics` を拡張）。|

### A-2. 非機能要件

| 要件 | 内容 |
| :--- | :--- |
| **セキュリティ** | クレジットカード情報は一切保持しない（Stripe に完全委譲）。Supabase RLS による行レベルアクセス制御。 |
| **拡張性** | 新プロダクト追加時は、Stripe に商品を追加し、LP ページを 1 枚作るだけで対応可能な設計。 |
| **パフォーマンス** | Next.js Server Components (RSC) による高速レンダリング。ダッシュボードの初期表示 < 1秒。 |
| **デプロイ** | Cloudflare Pages。`output: 'export'` を**解除**し、SSR モードへ移行する必要あり（Webhook 受信のため）。 |

### A-3. 非要件: 「何を・何で実装しないか」

| やらないこと | 理由 |
| :--- | :--- |
| **自前の決済フォーム** | PCI DSS 準拠コストが膨大。Stripe Checkout に完全委譲。 |
| **自前のパスワード認証** | セキュリティホールのリスク。Supabase Auth に完全委譲。 |
| **SNS/コミュニティ機能** | プロダクトの核心ではなく、開発コストに見合わない。 |
| **組織内の複数ユーザー管理 (RBAC)** | 初期フェーズでは 1 アカウント = 1 契約。複雑な権限管理は後回し。 |
| **LLM へのリクエスト中継** | ポータルの責務ではない。ゲートウェイサーバーの責務。 |
| **デスクトップアプリのホスティング/配布** | GitHub Releases や別途インストーラーで対応。 |

## B. Ayato Auth & API Gateway (Server / 新規構築)

**リポジトリ**: 新規作成 (例: `ayato-cloud-gateway`)
**技術スタック**: Python (FastAPI or Flask) / Docker / Cloud Run (GCP)
**データソース**: Supabase (トークン検証用に読み取りのみ)

### B-1. 機能要件: 「何を・何で実装するか」

| 機能 | 使用技術 | 説明 |
| :--- | :--- | :--- |
| **トークン認証** | **Supabase Client (Python)** | リクエストヘッダーの `Authorization: Bearer ayato-xxxx` を受け取り、`api_tokens` テーブルで有効性と `plan_type` を検証。 |
| **LLM プロキシ** | **google-genai SDK** / **httpx** | 有効なリクエストを Gemini, Vertex AI 等へ中継し、レスポンスを返す。 |
| **レートリミット** | **メモリ内カウンター** or **Redis** | `plan_type` に基づき、月間/日次の API コール回数を制限。 |
| **使用量記録** | **Supabase DB** (`organizations.current_usage_count` を更新) | API コール 1 回ごとに使用量をインクリメント。 |
| **ヘルスチェック** | **`/health` エンドポイント** | Cloud Run のヘルスチェック用。 |

### B-2. 非機能要件

| 要件 | 内容 |
| :--- | :--- |
| **セキュリティ** | ユーザーのプロンプト/レスポンスをサーバー側に永続保存しない（ステートレス中継）。SSL/TLS 必須。|
| **スケーラビリティ** | Cloud Run のオートスケーリングに依存。ステートレス設計。 |
| **コスト管理** | ユーザーのサブスク料金内で逆ザヤにならないよう、モデル別のコスト計算ロジックを持つ。 |
| **可用性** | Cloud Run のマネージド基盤に依存。SLA 99.95%。 |

### B-3. 非要件: 「何を・何で実装しないか」

| やらないこと | 理由 |
| :--- | :--- |
| **ユーザー登録/ログイン機能** | ポータル (Supabase Auth) の責務。ゲートウェイはトークンの真偽だけを検証する。 |
| **決済処理** | ポータル (Stripe) の責務。ゲートウェイは `plan_type` を参照するだけ。 |
| **Web UI の提供** | 一切の HTML/CSS を返さない。JSON API のみ。 |
| **プロンプト/レスポンスの永続保存** | プライバシー原則。ログは匿名化された使用量のみ。 |
| **デスクトップアプリの配布** | 別リポジトリの責務。 |

## C. Ayato Transcriber (Desktop App / 既存)

**リポジトリ**: `Transform_MovieToText`
**技術スタック(既存)**: Python / Flet / Whisper / Ollama / SQLite
**配布**: Windows exe (PyInstaller)

### C-1. 機能要件: 「何を・何で実装するか」(追加分)

| 機能 | 使用技術 | 説明 |
| :--- | :--- | :--- |
| **クラウドトークン入力** | **Flet UI** (TextField) | SettingsView にトークン入力欄を設置。**実装済み**。 |
| **トークンによるエディション判定** | **ConfigManager** | トークンが有効なら `PRO` としてローカル制限を解除。**実装済み (Mock)**。 |
| **Gateway 経由の LLM 通信** | **AyatoCloudClient** (requests/httpx) | `ayato_cloud` プロバイダーとして Gateway にリクエスト送信。**実装済み (Stub)**。 |
| **ポータルへの誘導** | **Flet UI** (Link/Button) | 「アカウント作成」「Pro にアップグレード」ボタンからブラウザでポータルを開く。 |

### C-2. 非機能要件

| 要件 | 内容 |
| :--- | :--- |
| **オフライン動作** | Free 版の全機能（ローカル Whisper + Gemma）はネットワーク不要で動作し続ける。 |
| **トークンキャッシュ** | 一度認証成功したトークンは、X 日間はオフラインでも PRO 機能を維持する。 |
| **プライバシー** | Free 版では一切の外部通信を行わない。Pro 版でも Gateway 以外とは通信しない。 |

### C-3. 非要件: 「何を・何で実装しないか」

| やらないこと | 理由 |
| :--- | :--- |
| **アプリ内での決済処理** | ポータルの責務。アプリからは「ポータルへのリンク」のみ提供。 |
| **アプリ内でのアカウント作成** | ポータル (Supabase Auth) の責務。 |
| **LLM API の直接呼び出し (Pro版)** | Gateway 経由に統一。BYOK は上級者向けの隠しオプションとして残す。 |
| **クレジットカード情報の入力** | 絶対にやらない。Stripe + ポータルの責務。 |

## 実装の優先順序 (ロードマップ)

```
Phase A: Portal (認証・決済基盤)
  A-1. Google ログイン (Supabase Auth)
  A-2. Stripe 統合 (Checkout + Webhook)
  A-3. マイページ (トークン発行 UI)
       |
Phase B: Gateway (API 中継サーバー / 新規)
  B-1. トークン認証 API
  B-2. LLM プロキシ (Gemini 転送)
  B-3. レートリミット
       |
Phase C: Desktop App (最終接続)
  C-1. Gateway への実通信に切替
  C-2. ポータルリンクの UI 追加
  C-3. 結合テスト・リリース
```

## 検討事項 (Open Questions)

- **Cloudflare Pages から Cloudflare Workers / Vercel への移行**: Webhook (POST) を受けるには SSR が必要。現在の `output: 'export'` は解除が必要。
- **Gateway のドメイン**: `api.ayato-ai.com` を取得するか、Cloud Run のデフォルト URL を使うか。
- **トライアル期間**: 新規ユーザーに 7 日間の Pro 体験を自動付与するか。
- **同時接続制限**: 1 トークンにつき何台のデバイスからの同時利用を許可するか。
