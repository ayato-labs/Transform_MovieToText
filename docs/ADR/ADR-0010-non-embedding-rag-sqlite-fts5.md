# ADR-0010: Non-Embedding RAG (SQLite FTS5) の採用とベクトル検索の廃止

- **Date**: 2026-06-01
- **Status**: Accepted
- **Deciders**: AI Agent & ayato-labs

## Context
本アプリケーションは、ローカルPC上で動作するデスクトップアプリとして配布され、完全オフライン環境で機密性の高い会議を処理し、過去のナレッジを検索できることを目指しています。

過去のバージョンでは、ナレッジ検索（RAG: Retrieval-Augmented Generation）基盤として `fastembed` 等のEmbeddingモデルとベクトルデータベースを導入していました。しかし、以下の課題に直面しました。

1. **VRAMの枯渇**: WindowsにおけるローカルAI環境では、ユーザーマシンのVRAM（メモリ）容量が最大のボトルネックとなります。LLM（Ollama等）とWhisperモデルだけでメモリを圧迫している状態において、さらにEmbeddingモデル（数GB）をメモリに常駐させるのはOOM（メモリ不足エラー）の直接的な原因になります。
2. **DLL依存とファイルサイズ**: C++ベースのEmbeddingライブラリやベクトルデータベース（Chroma等）をPyInstallerで単一EXEに固める際、複雑なDLL依存関係により環境構築が失敗しやすく、配布バイナリのサイズも不要に膨れ上がります。
3. **検索レイテンシ**: 検索のたびにクエリをベクトル化するオーバーヘッドがあり、ユーザー体験（体感レスポンス）を損なっていました。

## Decision
Embeddingモデル（ベクトル化）およびベクトルデータベースへの依存を完全に排除し、**「LLMによるタグ抽出」と「SQLite FTS5」を組み合わせた疑似RAG（Lexical Search）アーキテクチャを採用**しました。

1. 会議の文字起こし完了時、要約プロンプトと並行して、LLM（`extract_category`）に「会議の内容から適切なカテゴリ（タグ）を1〜3つ抽出」させます。
2. 抽出したタグ、議事録、文字起こし本文を、Windows環境の標準ライブラリで動作する `SQLite FTS5` (Full Text Search) テーブルに格納します。
3. ナレッジ検索時には、ベクトル空間の探索ではなく、BM25アルゴリズムベースの高速なキーワード全文検索（MATCH構文）を実行します。

## Consequences

### Positive
- **VRAM/RAM消費の劇的な削減**: 巨大なEmbeddingモデルのロードが不要になり、ローカルPCのリソースをLLMと音声認識（Whisper）に全振りできるようになりました。
- **配布と保守の容易性**: サードパーティ製のベクトルDBへの依存が消え、単一EXEによる配布が極めて安定し、バイナリサイズも縮小しました。
- **検索の爆速化**: ミリ秒単位で完了する超高速なレイテンシを実現しました。

### Negative / Risks
- **セマンティック検索の喪失**: 単語の「意味の近さ」による検索ができないため、表記揺れ（例：「PC」と「パソコン」）や、曖昧な文脈での検索には弱くなります。
- **LLM依存の抽出漏れ**: LLMが生成するタグが不適切だった場合や、議事録の要約から重要なキーワードが欠落した場合、全文検索でヒットしなくなるリスクがあります。

### Future Outlook (将来の展望)
現在のLexical Search（単語一致）アプローチで検索精度に限界が来た場合（行き詰まった場合）や、より高度な概念検索のニーズが高まった場合には、**数百MB程度の超軽量なEmbeddingモデルを限定的に再導入し、FTS5と組み合わせたハイブリッド検索へ移行することも視野に入れています**。今回の決定は「永久にEmbeddingを使わない」というものではなく、現在の「配布容易性とVRAM制約」というフェーズに最適化した戦略的なトレードオフです。
