# 概念的な機能要件・非機能要件

## プロダクト戦略と開発哲学（Strategic Philosophy）
**「まずは『ローカル完結・完全無料』で圧倒的な認知と信頼を獲得する」**

本プロジェクト（Transform_MovieToText）は、SaaS型の収益化やGatewayサーバーの構築といった複雑なマネタイズを一旦棚上げし、**「100%ローカルのOllama依存」という純粋な無料デスクトップアプリ**として初期のトラクション（ユーザーの需要）を捉えることに全振りします。

*   **個人開発の最大の強み**：巨大なインフラ維持費や「すぐに収益化しなければならない」という株主からのプレッシャーがありません。
*   **無料という最強のフック**：「1バイトも外部に送信されない」「商用SaaSなら月額数千円かかるレベルの高精度なAI文字起こしと要約が、自分のPCで完全無料・無制限に使える」という圧倒的な価値を提供し、熱狂的なコミュニティと知名度を獲得します。
*   **収益化は『その先』にある**：まずはペルソナ（需要）を確実に取り込みます。ビジネス利用における「BYOK（個人APIキー）の解放」や、「クラウド連携によるさらに高度な自動化」といった有料パッケージ化は、ユーザーが本アプリを手放せなくなってから（十分な信頼を構築した後）提示するのが、最も合理的で持続可能な戦略です。

そのため、現在のアーキテクチャでは **「Ollamaのローカル推論（`cloud`モデルの厳格な除外）」** にのみ絞り込み、ユーザーに「1滴のデータ流出リスクもない」という究極の安心感と利便性を提供します。

## 主幹の提供価値
- ローカル完結で、**情報漏洩リスクが限りなくゼロ**であること。

## 文字起こしのチャンクのつなぎ目について
５秒ほど音声ファイルを物理的に重ねることで、切れ目をなくした。
文字起こしが一時的に重複したり不自然になる箇所が発生するが、最終的に人間が参照するのは「LLMが構造化した議事録」であるため、無音検知の複雑な処理を入れるよりも「物理カット＋重ね合わせ」による安定性を優先・許容する。

## AI/推論エンジンに関する要件
- **量子化・混合精度推論**: VRAM使用量を削減しつつ、推論速度を向上（コストパフォーマンスの最大化）。
- **Ollamaローカル連携**: 使用するLLMモデルは、Ollamaライブラリを使用してローカルホスト（`http://localhost:11434`）から動的にモデル一覧を取得しUIの選択肢とする。
  - **重要制約**: Ollamaが提供を開始したCloudサービスに由来するモデル（名前に`cloud`を含むもの）は、完全ローカルの信念に反するため、一覧取得時にシステム側で強制的かつ自動的に除外する。
- **ハードウェア自動最適化**: 実行マシンのスペック（RAM/VRAM等）を自動検知し、最適なローカルAIモデルや推論設定（CPUフォールバック等）を調整して動作させる。

## 必要なコア機能
- **ファイルからの文字起こし**: 既存の動画/音声ファイルからテキストを抽出。
- **リアルタイム文字起こし**: 会議等の音声をリアルタイムでキャプチャし文字起こし。
- **自動プロセッシング (AI議事録化)**: 文字起こし完了後、ローカルLLMを用いて自動的に清書、要約、議事録の構造化フォーマットへ変換。
- **RAG (Retrieval-Augmented Generation) チャットボット**: 過去の会議録や文字起こしデータに対して自然言語でチャット検索が可能。
  - チャット応答には、「どのタイトルの情報を参考にしたか」のソース（出処）をプロンプトエンジニアリングではなく、システムロジックとして自動挿入・明記する。
- **検索アーキテクチャの最適化（Non-Embedding RAG）**:
  - **意図的非採用**: `fastEmbed` やベクトルデータベース等のEmbedding（ベクトル化）モデルは使用**しない**。ベクトル検索は「なんとなく似た文脈」を拾うのには適しているが、ローカルでの計算コスト（RAM/VRAMの圧迫）が高く、個人開発のデスクトップツールとしてはオーバースペックとなりやすい。
  - **代替アプローチ**: **SQLite FTS5 (Full-Text Search) + LLM Intent Analysis（意図解析）** のハイブリッド手法を採用。
    1. ユーザーの質問から、LLMが「検索すべきキーワード」「対象プロジェクト」「対象カテゴリー」を超軽量に抽出。
    2. 抽出されたメタデータとキーワードを用いて、SQLiteのFTS5機能で高速に絞り込み検索（ハードウェア負荷はほぼゼロ）。
    3. ピンポイントで絞り込まれた文書のみをLLMに渡し、最終的な回答を生成する。
  - この手法により、NotebookLMのような重厚なベクトルRAGと同等の「ナレッジ引き出し」を、ローカルPCの限られたリソースで極めて軽快に実現する。
- **AI自動タグ生成機能**: LLMを用いて内容からタグを自動生成し、メタデータとしてSQLiteデータベースに保存。
- **プロジェクト管理機能**:
  - ファイル録音、リアルタイム録音ともに「プロジェクト」を付与・整理できる機能。
  - 選択肢には常に「その他」がデフォルト存在し、「新規プロジェクト」を選択した時のみ名前入力用のテキストボックスを出力するUIUX設計。
  - 履歴UIにおけるプロジェクトでの柔軟な絞り込み（フィルター）および解除。
  - 設定UIからのプロジェクト削除機能。削除されたプロジェクトに属していたデータは、自動的に「その他」へ再割り当てされるセーフフォールバック。
- **分散型ナレッジ共有機能 (Personal-to-Individual)**:
  - **エクスポート**: 任意の会議や文書を、人間が読みやすいMarkdown（`.md`）形式で外部へ書き出し可能。
  - **メタデータ保持 (YAML Frontmatter)**: 書き出されたファイルには「タイトル」「プロジェクト」「カテゴリー」「日付」などのメタデータがYAMLヘッダーとして埋め込まれ、データとともに「文脈」も可搬性を持たせる。
  - **P2P共有戦略**: クラウド共有サーバーを持たず、書き出したファイルをDropbox, OneDrive, NAS等の共有フォルダに配置するだけで、他ユーザーの `KnowledgeScanner` が自動的にそのナレッジを吸収・インポートする「Gitスタイル」の分散共有を実現。
  - **情報の所有権**: 共有前にテキストエディタで内容を編集・検閲できるため、ローカル完結のプライバシーポリシーをチーム共有時でも維持可能。

### 技術的メモ・課題方針
- **音声カット**: 物理カット ＋ LLM接合（無音検知よりも安定性が高くローカル環境に最適であるため現状維持）。
- **暗号化についてのスタンス**:
  - ハードウェアID暗号化: 物理的盗難には強いが、オープンソース環境では気休めになりがち。
  - リカバリーの課題: ローカル暗号化には「パスワード忘れ」による不可逆なデータ紛失リスクが付き物であり、優先度は下げる。

## 第2期 実装機能（2026年4月アップデート）
- **Map-Reduce型 長文要約ロジック**: 
  - 長時間の会議（1万文字以上など）の文字起こしデータに対し、Gemma 3 (2B/4B) などのコンテキスト長制限を克服するため、自動で意味の切れ目（チャンク）に分割。
  - 各チャンクの中間要約（Map）を行い、最後に統合して最終議事録（Reduce）を生成することで、長文でも高精度な要約を維持。
- **AIモデル・ライフサイクル管理 (Garbage Collection)**:
  - UIの「設定」画面から、PC内に保存されているOllamaのローカルAIモデル一覧とそのディスク使用量（GB）を確認可能。
  - 不要になった古いモデルをワンクリックで安全に削除（アンインストール）し、ユーザーのストレージをクリーンに保つ機能。
- **トランスペアレンシー・ダッシュボード (進捗の可視化)**:
  - バックグラウンドでのAIモデルのダウンロード（ollama pull）時に、Ollama APIのストリーミング通信を利用。
  - ダウンロード済みのデータ量（MB）と全体の進捗率を、ブート画面およびメイン画面のステータスバーにリアルタイムかつ正確に描画し、ブラックボックス化を排除。
- **Zero-Trust Local API (ローカルAPIの堅牢化)**:
  - アプリ起動時に環境変数 \OLLAMA_HOST=127.0.0.1:11434\ をアプリケーション全体に強制注入。
  - OllamaのPython SDKおよびCLIからの通信がすべてループバックアドレス（127.0.0.1）に固定され、外部ネットワークからの不正アクセスや意図しないクラウドモデルへのデータ流出を物理的・ネットワーク的に100%遮断。
