# RAG Intent Extraction Strategy for 1B Models (Gemma 3 1B)

## 背景
ローカル環境(VRAM制限環境)で RAG (Retrieval-Augmented Generation) を構築する場合、意図抽出(クエリ分析)に Gemma 3 1B などの極小モデルを使用することが不可欠。
しかし、1Bクラスのモデルには「論理的推論」や「構造化データ(JSON)の厳密な出力」に限界があることが判明した。

## 課題
- **JSON出力の不安定性**: 複雑なプロンプトを与えると、JSONフォーマットが崩れる、あるいは空の結果を返す。
- **否定/訂正の誤認**: 「AではなくてB」という自然な発話において、AとBの両方を抽出してしまう、あるいは否定を理解できない。
- **低IQ(パラメータ不足)**: 推論ステップが多くなると、途中で注意力が散漫になり、重要なキーワードを落とす。

## 解決策: Ultra-Minimal Intent Extraction Strategy
極小モデルには「論理」を求めず、「リストからの識別」のみを任せる。

### 1. 超簡略化プロンプト (Ultra-Minimal Prompt)
モデルに対し、JSON形式の指定や「〜以外」などの複雑な条件判断を指示せず、以下の単純なタスクのみを課す。
- 「提示されたプロジェクト/タグのリストから、関連するものを箇条書きで抜き出せ」

### 2. プログラマティック・パース (Regex & Fuzzy Match)
モデルの生出力をコード側で後処理する。
- **正規表現 (Regex)**: LLMの出力テキストから直接プロジェクト名/タグ名をパターンマッチングで抽出。
- **曖昧マッチ (Fuzzy Matching)**: `difflib` 等を用いて、微細なタイポや略称をシステム内の正解データに自動補正。
- **類義語マップ (Synonyms)**: 「要約」「議事録」「分」などを `Minutes` に統一する辞書をプログラム側に持たせる。

## 結論
1Bモデルにおいては、**「モデルに構造を強いるのではなく、コード側で構造化する」**というアプローチが、最も高い堅牢性と再現性を生む。
本プロジェクトの RAG 実装は、この方針に基づいて `QueryAnalyzer` を構築する。
